# Contributing to Grafana Pathfinder

Thanks for taking the time to contribute. Feedback, issues, and pull requests are all welcome, whether it's a typo fix, a new interactive guide, or a feature.

This guide covers what we look for in a pull request. When in doubt, open the issue or PR anyway and ask. We'd rather help you get a contribution over the line than have it stall on process.

## Ways to contribute

- **Report a bug**: open an issue. For problems with an interactive guide, the recording-bug issue template helps us reproduce it.
- **Suggest an improvement**: open an issue describing the problem you're trying to solve before writing code, especially for anything user-facing or architectural. It saves you building something we'd ask you to rework.
- **Fix a bug or add a feature**: open a pull request. Read the rest of this guide first.

## Signed commits

This repository requires every commit to carry a verified signature, and your PR's checks will not pass unless all of its commits are signed. This is a recent requirement, so it catches people out. If you haven't set up commit signing before, GitHub's guide walks through it: [signing commits](https://docs.github.com/en/authentication/managing-commit-signature-verification/signing-commits).

## Contributor License Agreement

You must sign the Contributor License Agreement (CLA) before we can merge an external PR. Signing is a step in the build process, so check the checks on your PR for a link to the CLA if you haven't signed it already.

## Pull request scope

**Keep each pull request focused on a single concern.** This is the one guideline that most affects how quickly your work gets merged, so it's worth a short explanation of why.

A focused PR is easier and faster to review, can be merged the moment it's ready instead of waiting on unrelated work, gets a clean changelog entry, and can be reverted on its own if a problem turns up later. A PR that bundles several unrelated changes has to be reviewed and merged as all-or-nothing, so one open question on any part holds up the rest.

This repository will likely reject large, unfocused PRs that combine too many concerns, or that show clear evidence of being auto-generated by an agent without human review. A good rule of thumb is to keep a PR to roughly 500 lines of changes or fewer, so it stays human-sized for review.

In practice:

- **One bug fix or one feature per PR.** If you find yourself writing "and also" in the description, that's usually a sign it should be two PRs.
- **Closely-related changes can share a PR.** A fix plus its test, or a handful of small changes to the same area that only make sense together, are fine in one PR.
- **Unrelated changes should be separate PRs**, even when they're small and you spotted them at the same time.
- **Use your judgment, and lean toward separate when unsure.** The goal is reviewable, revertable units of change, not the largest or smallest possible PR. Don't split a single coherent change into fragments just to have more PRs.

If you're partway through and realise a change is growing beyond one concern, it's completely fine to open a draft PR and ask how best to split it.

## Before you open a pull request

- **Run the full check.** `npm run check` runs the same gate as CI (typecheck, lint, prettier, Go lint, Go tests, and frontend tests with coverage). PRs that pass locally move faster.
- **Add or update tests** for the behaviour you're changing.
- **Match the existing style.** All UI text and documentation uses [sentence case](https://grafana.com/docs/writers-toolkit/write/style-guide/capitalization-punctuation/#capitalization): capitalise only the first word and proper nouns. Keep comments to what the code can't say for itself.
- **Link the issue(s)** your PR addresses in the description.
- **Write a clear description** of what changed and why. The "why" is the part reviewers can't reconstruct from the diff.

## Review

A maintainer will review your PR and may suggest changes. Review comments are about the code, not about you, and a request to split or refocus a PR isn't a rejection of the contribution, just a way to get each part merged sooner. We aim to be responsive; if a PR goes quiet, a polite nudge is welcome.
